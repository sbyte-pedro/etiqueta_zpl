# Deployment Design — Railway + Postgres
**Date:** 2026-07-22
**Status:** Approved

---

## Context

The Zebra Label Designer is currently a local-only monorepo (React frontend + Express backend + SQLite). The goal is to deploy it to a public URL for demo sharing (C), with an architecture that scales to production use by warehouse operators (D) without a second major migration.

---

## Goals

- Public HTTPS URL shareable with anyone
- Backend connected to managed Postgres (replaces SQLite)
- Frontend served as a static site with backend URL injected at build time
- All secrets in environment variables, no hardcoded values
- Zero frontend component changes required

---

## Non-Goals

- Multi-region / CDN setup (post-v1)
- CI/CD pipeline (manual deploy from Railway dashboard is fine for now)
- Custom domain (Railway-generated URL is sufficient for demo)
- Container orchestration (Kubernetes, ECS)

---

## Target Architecture

```
Railway Project
├── Service: backend
│   ├── Source: /backend (Nixpacks detects Node, runs tsc + node dist/index.js)
│   ├── Env vars: JWT_SECRET, FRONTEND_URL, PORT (auto), DATABASE_URL (auto)
│   └── Plugin: Postgres (Railway managed — injects DATABASE_URL)
│
└── Service: frontend
    ├── Source: /frontend (Nixpacks runs vite build, serves dist/)
    └── Env vars: VITE_API_URL (set to backend service URL)

External: api.labelary.com (outbound HTTP from backend, no changes needed)
```

## Code Changes

### 1. Replace SQLite with Postgres

**File:** `backend/src/db/database.ts`

- Remove `better-sqlite3` dependency
- Add `drizzle-orm` + `drizzle-orm/node-postgres` + `pg`
- Add `drizzle-kit` as devDependency for migrations
- Schema stays identical: `users` table, `designs` table
- Add `drizzle.config.ts` at repo root pointing to `backend/src/db/schema.ts`
- Add `backend/src/db/schema.ts` defining tables with Drizzle syntax
- Add `backend/drizzle/` migration folder (generated by `drizzle-kit generate`)
- Run migrations on startup: `migrate(db, { migrationsFolder: 'drizzle' })`

**Services to update:** `authService.ts`, `designsService.ts` — replace raw SQLite calls with Drizzle query builder. Logic unchanged; only the driver syntax changes.

### 2. Backend: environment variables

**File:** `backend/src/index.ts` (or `app.ts`)

```ts
// Fail fast at startup if required env vars are missing
const JWT_SECRET = process.env.JWT_SECRET
if (!JWT_SECRET) throw new Error('JWT_SECRET env var is required')

const DATABASE_URL = process.env.DATABASE_URL
if (!DATABASE_URL) throw new Error('DATABASE_URL env var is required')

const PORT = process.env.PORT ?? '3001'
const FRONTEND_URL = process.env.FRONTEND_URL ?? 'http://localhost:5173'
```

**File:** `backend/src/app.ts` — update CORS:

```ts
cors({ origin: process.env.FRONTEND_URL ?? 'http://localhost:5173' })
```

### 3. Frontend: API base URL

**Files:** `frontend/src/utils/zplClient.ts` and any other files that call `fetch('/api/...')` directly (e.g. auth and designs API calls in the frontend)

- Replace hardcoded `/api` (which relied on Vite proxy) with:

```ts
const API_BASE = import.meta.env.VITE_API_URL ?? ''
// e.g. fetch(`${API_BASE}/api/generate-zpl`, ...)
```

- In dev, `VITE_API_URL` is unset → falls back to `''` → Vite proxy still handles `/api` → no change to dev workflow
- In production, `VITE_API_URL=https://backend-xxx.railway.app` → direct calls

**File:** `frontend/.env.example`

```
VITE_API_URL=https://your-backend.railway.app
```

### 4. Railway configuration

**File:** `backend/railway.toml`

```toml
[build]
builder = "NIXPACKS"

[deploy]
startCommand = "npm run build && node dist/index.js"
healthcheckPath = "/health"
healthcheckTimeout = 30
```

**File:** `frontend/railway.toml`

```toml
[build]
builder = "NIXPACKS"
buildCommand = "npm run build"

[deploy]
staticDir = "dist"
```

**File:** `backend/src/routes/health.ts` — add a simple health endpoint:

```ts
router.get('/health', (_, res) => res.json({ ok: true }))
```

## Environment Variables Reference

| Service  | Variable        | Source                    | Notes                              |
|----------|-----------------|---------------------------|------------------------------------|
| backend  | `DATABASE_URL`  | Railway Postgres plugin   | Auto-injected, no manual action    |
| backend  | `JWT_SECRET`    | Manual (Railway dashboard)| Generate with `openssl rand -hex 32` |
| backend  | `FRONTEND_URL`  | Manual (Railway dashboard)| Set to frontend service public URL |
| backend  | `PORT`          | Railway runtime           | Auto-injected                      |
| frontend | `VITE_API_URL`  | Manual (Railway dashboard)| Set to backend service public URL  |

## Data Migration (existing SQLite → Postgres)

If there is existing data in the local `.db` file worth preserving:

1. Export: `sqlite3 backend/data.db .dump > backup.sql`
2. Adapt the dump (SQLite types → Postgres types if needed)
3. Import: `psql $DATABASE_URL < backup.sql`

For a fresh demo with no production data, this step is skipped.

## Local Dev — No Change

The dev workflow is unchanged:

```bash
npm install
# Terminal 1
cd backend && npm run dev    # Express on :3001 (uses local .env with DATABASE_URL pointing to local Postgres or SQLite in dev)
# Terminal 2
cd frontend && npm run dev   # Vite on :5173, /api proxied to :3001
```

For local dev with Postgres: add a `backend/.env` with `DATABASE_URL=postgresql://localhost/etiqueta_dev`.
For local dev keeping SQLite: drizzle supports a SQLite adapter too — use it in dev, Postgres in prod (same schema, different adapter). This is optional but simplifies local setup.

## Deployment Steps (one-time)

1. Push repo to GitHub
2. Create Railway project → "Deploy from GitHub repo"
3. Add Postgres plugin to backend service
4. Set env vars: `JWT_SECRET`, `FRONTEND_URL`, `VITE_API_URL`
5. Deploy backend → confirm `/health` returns `{ ok: true }`
6. Deploy frontend → confirm app loads and `/api` calls reach backend
7. Test: register user, create label, save design, reload — confirm persistence

---

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Labelary API unavailable | Backend proxy returns HTTP error; frontend shows error state (already implemented) |
| `JWT_SECRET` missing in prod | Startup throws — Railway marks deploy as failed, no silent breakage |
| CORS blocks frontend | `FRONTEND_URL` env var is explicit; misconfiguration surfaces immediately on first API call |
| Schema drift between dev and prod | Drizzle-kit migration files are committed; run on startup automatically |
| SQLite data loss during switchover | One-time export script before migration; fresh demo needs no migration |
